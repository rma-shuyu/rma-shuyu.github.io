dpdk-vdpa热迁移

发表于 2022-06-01 | 分类于 DPDK |

设备状态保存

这里需要了解哪些状态信息需要保存， 主要涉及两部分：

a) 设备启动过程中以及启动之后前端驱动对 BAR 空间的配置和修改。 Vdpa 框架中， virtio 设备配置空间的仿真由 qemu 实现，因此设备状态的保存和恢复由 qemu 实现，并在目的端 vmstats_load 之后， vm_start 流程中调用 vdpa 接口下发 load 后的配置信息，启动设备。

b) 队列当前的头尾指针信息 , virtio 设备涉及 avail head 和 used tail 。 Vdpa 中，队列的状态信息在停止设备过程中调用 vdpa 接口从设备侧读取。

设备队列tx/rx 方向处理

队列当前的头尾指针信息， virtio 设备侧涉及 avail head 和 used tail 的保存和恢复。

a) 发送方向状态保存 :

保存 used tail, 恢复的时候将 avail head 和 used tail 设置成保存的 used tail 。如果迁移前 used tail < avail head ，迁移后目的端会重复发送部分数据（ used tail -> avail head ）， TCP seq/ack保障可靠性， UDP 侧收到重复数据可能会有影响。

b) 接收方向状态保存：

保存 used tail, 恢复的时候将 avail head 和 used tail 设置成保存的 used tail ，迁移后目的端设备会从 avail 队列中下拉 used tail -> avail tail 的数据。

vdpa内核态

legacy

1、源端设备 status协商至 0xb；
2、驱动给源端设备所有队列的 last_avail_idx寄存器写入值（按照流程，驱动不应写源端设备的last_avail_idx寄存器）；
3、驱动遍历此源设备的所有队列，置上 queue_msix_vector后 status协商至 0xf；
4、目的设备开始协商，协商 status至 0x3,表明驱动知晓如何驱动目的设备；
5、驱动写入源设备的脏页地址与大小寄存器：log_base_l、log_base_h、log_size_l以及 log_size_h；
6、驱动将 1写入源设备的 lm_ctrl寄存器，表明源设备使能热迁功能开始进行热迁，即 log start；
7、驱动读取源设备的 lm_ctrl，读到的值为 1，再将 2写入源设备的 lm_ctrl寄存器，表明源设备关停设备,即 dev stop；
8、驱动读取源设备每个队列的 last_avail_idx；
9、驱动写 0xb至目的设备的 status寄存器；
10、驱动从源端设备中读取每个队列的 last_avail_idx寄存器写至目的设备的对应的队列的last_avail_idx寄存器；
11、驱动遍历目的设备的所有队列，置上 queue_msix_vector后 status协商至 0xf；
12、驱动将 0写入源设备的 lm_ctrl寄存器，表明源设备 log stop；
注：前三步为在 VDPA的框架下先协商成功源设备；第四步为驱动发现并知晓如何驱动目的设备；第五步为了后面同步脏页，先将脏页的地址以及大小通知后端 vdev；第六步为 log start，vdev通知 admin开启脏页（根据第四步驱动写入的脏页地址与大小信息）；第七步为关停源设备（主要为 disable队列，同步脏页，spdk、dpdk停止设备）；第八步为驱动获取源设备的last_avail_idx；第九步为驱动协商目的设备的特性完毕；第十步为驱动将源设备的 last_avail_idx设置到目的设备中；第十一步为目的设备协商成功（vdev在协商成功后初始化硬件队列）；第十二步为源设备关闭脏页。
